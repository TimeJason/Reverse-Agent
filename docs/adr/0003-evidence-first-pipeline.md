# ADR 0003：采用 Evidence-first 管线与单向依赖

## 状态

已接受，2026-06-09。

## 背景

软件分析结果可能包含推断、不确定性和多来源关联。如果采集器直接输出结论、导出器重新解析原始数据，或接口层绕过应用服务访问数据库，同一输入会产生难以复现和难以审计的结果。

## 决策

核心分析管线固定为：

```text
Evidence -> Facts -> Findings -> Artifacts
```

各层职责：

- **Evidence**：可寻址、可校验的原始或标准化观测记录。
- **Facts**：从 Evidence 确定性提取的结构化事实。
- **Findings**：由 Evidence 和 Facts 支撑、带置信度的分析结论。
- **Artifacts**：面向交付的 OpenAPI、文档、集合或其他版本化产物。

依赖与访问规则：

- Provider 只能产生或导入 Evidence，不能直接产生 Finding。
- Facts 和 Findings 必须保留 `evidence_refs`。
- Exporter 消费结构化 Facts/Findings，不重新解析 raw evidence。
- MCP、CLI 和未来 Web/API 通过应用服务访问能力，不直接访问 SQLite。
- 核心领域不依赖 MCP SDK、数据库驱动或具体 Provider。
- LLM 只能作为可选 enrich 阶段，输出仍需证据引用；LLM 不得修改 raw evidence。

## 影响

正面影响：

- 每个结论都有明确来源，可以复现、解释和审计。
- 采集、分析和导出可独立演进与测试。
- 同一份证据可以重复运行不同版本管线，而无需重新采集。
- 外部工具接入不会绕过平台的数据与安全边界。

代价：

- 需要维护中间模型、引用完整性和管线运行元数据。
- 简单功能也必须经过分层，初期代码量高于一次性脚本。
- Schema 演进和缓存失效需要显式策略。

## 替代方案

- **Provider 直接生成报告**：交付快，但供应商输出不可比较，结论难以追溯。
- **Exporter 直接读取原始证据**：减少中间模型，但会重复解析逻辑并产生不一致结果。
- **LLM 端到端解释原始数据**：灵活，但不可复现、成本不稳定且增加敏感数据暴露。

## 后续工作

- 为四层对象定义版本化 Schema 和引用完整性检查。
- 为管线运行记录输入版本、步骤版本、输出与错误。
- 在模块边界测试中禁止反向依赖和绕过应用服务。
