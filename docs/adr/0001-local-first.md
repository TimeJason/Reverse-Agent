# ADR 0001：采用 Local-first 架构

## 状态

已接受，2026-06-09。

## 背景

平台需要处理 HTTP 流量、日志、浏览器轨迹、认证信息和潜在个人数据。这些证据通常敏感且体积较大。第一阶段的主要使用者是本地开发者与 AI 编程工具，核心目标是先建立可复现的采集、分析和导出管线，而不是建设多租户 SaaS。

## 决策

第一阶段采用 Local-first 架构：

- 项目工作区、原始证据、数据库和产物默认保存在用户选择的本地目录。
- 核心采集、查询、分析和导出不依赖云服务或在线账户。
- 本地服务默认只绑定回环地址。
- 远程同步、团队协作和托管分析只能作为后续可选能力，不能成为本地流程的前置条件。
- 数据模型可以预留 workspace、user、role 和 audit 概念，但第一阶段不实现完整多租户系统。

## 影响

正面影响：

- 敏感证据默认不离开设备。
- 离线环境和受限网络环境仍可工作。
- 安装、备份、迁移和故障排查可以围绕一个明确的项目工作区进行。
- 早期工程可以专注证据管线，不被云端身份、计费和租户隔离分散。

代价：

- 用户需要自行管理本地磁盘、备份和访问权限。
- 跨设备同步和多人协作不会在第一阶段自然获得。
- 本地依赖、证书和进程管理必须有清晰的诊断能力。

## 替代方案

- **Cloud-first SaaS**：协作体验更直接，但会显著增加敏感数据治理、租户隔离和运营复杂度。
- **纯内存或临时项目**：实现简单，但无法满足可追溯、可复现和长期分析要求。

## 后续工作

- 定义项目工作区布局和备份边界。
- 实现本地路径、权限、磁盘空间与回环绑定检查。
- 任何远程能力都必须单独进行安全评审，并默认关闭。
